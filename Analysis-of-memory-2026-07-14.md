# TECHNICAL ANALYSIS OF PLUGIN: memory

## 1. Purpose & Architecture

- **Primary Function:** Persistent memory system for LM Studio. Stores facts globally (`global.md`) and per-conversation (`&lt;chatId&gt;_&lt;name&gt;.md`), auto-injects them into every prompt so the model retains context across sessions.
- **Entry Point:** `src/index.ts` → `main()` — registers config schematics, tools provider, and prompt preprocessor via `PluginContext`.
- **Component Map:**

| File | Responsibility |
|------|---------------|

| `src/toolsProvider.ts` | 6 tools: `read_memory_global`, `read_memory`, `write_memory_global`, `write_memory`, `remove_memory_global`, `remove_memory` |

## 2. Lifecycle & Data Flow

### Initialization
1. `main()` calls `bootstrapMemoriesDir()` → creates `~/.lmstudio/memories/`
2. Registers `configSchematics` (autoInject setting)
3. Registers `toolsProvider` (6 tools)
4. Registers `promptPreprocessor`

### Interaction Flow (Auto-Inject Path)
1. User sends message → `promptPreprocessor` fires
2. `parseRememberRequest` checks three regex prefixes:
   - `^remember\s+chat:\s*` → auto-save to chat memory
   - `^remember\s+global:\s*` → auto-save to global memory
   - `^remember:\s*` → model chooses scope (injects guidance note)
3. If no remember command and `autoInject=on`:
   - Reads global + chat memory files
   - Computes fingerprint (`chatFileName|globalRaw|chatRaw`)
   - Deduplicates injections via `stateMap` + 15-min interval
   - Injects memory block into prompt via `injectIntoMessage()`

### Interaction Flow (Tool Path)
1. Model calls a memory tool → `toolsProvider` implementation
2. `write_memory_global` / `write_memory` → `upsertMemoryLine` replaces or appends
3. `remove_memory_global` / `remove_memory` → `splice` by line number
4. All writes immediately re-read and return updated state

### Data Storage
- **Global:** `~/.lmstudio/memories/global.md` (one fact per line, `Label: value`)
- **Chat:** `~/.lmstudio/memories/&lt;chatId&gt;_&lt;name&gt;.md`
- **Legacy support:** Old `&lt;chatId&gt;.md` files get merged into new format

## 4. Security Analysis & Limitations

### 🔴 Critical: Broken Production Bundle — Plugin Won't Load

The `production.js` bundle contains this import:
```javascript
import("./../src/index.ts")
```
This is an **ESM dynamic import pointing to TypeScript source code** (`src/index.ts`), which does not exist at runtime. The `.lmstudio/production.js` was generated by `lms dev` (which bundles with ESM), but the entry.ts still references `src/index.ts`. The `build-production.sh` script attempts to fix this with `sed` replacements, but:

- The sed pattern `s|src/index.ts|dist/index.js|g` runs on the **already-bundled** `production.js` — but the bundle already has `./../src/index.ts`, not `src/index.ts`, so the sed **does not match**
- This is the exact error logged in `not_fixed.md`: `ERR_MODULE_NOT_FOUND: Cannot find module 'C:\Users\Zver\.cache\lm-studio\extensions\plugins\esalesninja\memory\src\index.ts'`
- **Result:** The plugin **fails to load** on first run unless manually patched

### 🟠 High: Build Script Is Fragile & Error-Prone

`build-production.sh` has multiple issues:
1. **Fuzzy sed-based path rewriting** — uses `sed -i` to replace paths in generated files. This is not a real build; it's a post-hoc hack
2. **Loops on failure** — if `lms dev --install` regenerates the entry, the script retries up to 3 times
3. **No verification of bundle validity** — only checks if `src/index.ts` string still exists (which it doesn't after bundling)
4. **Copies `dist/` to install dir** — but the bundle itself references source paths, creating a contradiction
5. **`set -euo pipefail`** is set, but the script does not `cd` to the plugin root before running `npm run build:production` — it does `cd "$ROOT"` which is correct, but the release script copies `dist/` recursively which could include `.map` source maps

### 🟠 High: No File Locking — Concurrent Write Corruption

`memoryStore.ts` reads and writes memory files **without any locking**:
```typescript
function writeGlobalMemory(content: string): MemoryFileContents {
  ensureMemoriesDir();
  const filePath = getGlobalMemoryPath();
  const lines = upsertMemoryLine(readLines(filePath), content);  // READ
  fs.writeFileSync(filePath, formatLines(lines), "utf-8");       // WRITE
  return buildMemoryContents(filePath, GLOBAL_MEMORY_FILE);
}
```
If two tools fire concurrently (e.g., `write_memory_global` + `remove_memory_global`), the second read could see stale data → **lost updates / index-out-of-bounds errors**.

### 🟡 Medium: `sanitizeFileName` Returns Empty String Silently

```typescript
function sanitizeFileName(name: string): string {
  const trimmed = name.trim();
  if (!trimmed) return "";  // ← Returns "" instead of throwing
  return trimmed
    .replace(/[/\\?%*:|"&lt;&gt;]/g, "-")
    .replace(/\s+/g, "-")
    .replace(/-+/g, "-")
    .replace(/^-+|-+$/g, "")
    .slice(0, 120);
}
```
If `name` is empty or all special chars, it returns `""`. Then in `buildMemoryFileName`:
```typescript
const slug = nameSlug || UNNAMED_CHAT_LABEL;  // Falls back to "unnamed"
```
This masks the issue — but it means a chat named `""` or `"   "` produces an `unnamed` slug, which could collide with genuinely unnamed chats.

### 🟡 Medium: `readConversationName` JSON Parse Too Broad

```typescript
function readConversationName(chatId: string): string | null {
  const conversationPath = path.join(CONVERSATIONS_DIR, `${chatId}.conversation.json`);
  try {
    if (!fs.existsSync(conversationPath)) return null;
    const data = JSON.parse(fs.readFileSync(conversationPath, "utf-8"));
    const name = typeof data.name === "string" ? data.name.trim() : "";
    return name || null;
  } catch {
    return null;
  }
}
```
- Catches **all** errors including `SyntaxError` from malformed JSON — silently returns `null`
- If `data` is not an object (e.g., `data` is `null` or an array), `data.name` throws — caught silently
- No validation that the file path is within expected boundaries

### 🟡 Medium: `MEMORY_KEY_PATTERN` Regex Accepts Anything Before First Colon

```typescript
export const MEMORY_KEY_PATTERN = /^([^:]+):\s*.+$/;
```
This matches **any string with a colon** — including potentially dangerous keys like `../../../etc/passwd: value` or `; rm -rf /: value`. The regex doesn't validate that the key is a "safe" label.

### 🟡 Medium: `upsertMemoryLine` Doesn't Validate Key Format

```typescript
export function upsertMemoryLine(lines: string[], content: string): string[] {
  const normalized = content.trim();
  if (!normalized) throw new Error("Memory content cannot be empty.");
  const newKey = getMemoryKey(normalized);
  if (!newKey) return [...lines, normalized];  // ← Appends line without key validation
  // ...
}
```
If a user saves `"no colon here"` as a memory, it's appended as-is. The system then can't parse it as a key-value pair later.

### 🟡 Low: `stateMap` Uses `unknown` Key Type

```typescript
const stateMap = new Map&lt;unknown, { fingerprint: string; injectedAt: number }&gt;();
```
Using `unknown` as the key type prevents TypeScript from catching typos in controller references. Should be `PromptPreprocessorController | ToolsProviderController`.

### 🟡 Low: `formatLines` Always Adds Trailing Newline

```typescript
function formatLines(lines: string[]): string {
  if (lines.length === 0) return "";
  return `${lines.join("\n")}\n`;
}
```
Even for a single-line file, this produces `"content\n"`. This means `readLines` will always trim the trailing newline, but `formatLines` always adds one — creating a subtle asymmetry. If a file is read, modified, and written back, the line count stays the same, but the raw content differs by one `\n`.

### 🟢 Low: No Error Propagation in `promptPreprocessor`

The outer `try/catch` in `promptPreprocessor` catches all errors and logs them with `console.warn`, then returns the original message. This means:
- Errors are **silently swallowed** from the user's perspective
- No error messages reach the model or UI
- `stateMap.set()` could fail if the controller key is invalid (though unlikely)

### 🟢 Low: `mergeMemoryFiles` Could Create Duplicate Lines

```typescript
function mergeMemoryFiles(sourcePath: string, targetPath: string): void {
  const merged = [...readNonEmptyLines(targetPath), ...readNonEmptyLines(sourcePath)];
  writeLines(targetPath, merged);
}
```
If the same label exists in both source and target files, it appears **twice** in the merged file. The upsert logic only works within a single file; cross-file duplicates are not resolved during merge.

## 6. Recommendations

1. **Fix the bundle import path:** The `production.js` must import `dist/index.js` (or use a relative path that resolves correctly at runtime). The current `./../src/index.ts` will never resolve.
2. **Replace `sed` post-processing with proper bundling:** Use a bundler (esbuild, webpack, or rollup) configured in `tsconfig.json` or a separate config to produce a single bundle with correct relative paths.
3. **Add file locking** for concurrent memory operations (e.g., `fs.flockSync()`).
4. **Validate memory keys** — restrict `MEMORY_KEY_PATTERN` to alphanumeric + spaces, reject paths and shell metacharacters.
5. **Improve `sanitizeFileName`** — throw or log when sanitization produces an empty string.
6. **Add `outFile` to tsconfig** for the release build to produce a single bundle, or configure esbuild for the production build.
7. **Add proper error handling** to `readConversationName` — distinguish `SyntaxError` from other errors.
8. **Fix `mergeMemoryFiles`** — run `upsertMemoryLine` on the merged result to deduplicate labels.
